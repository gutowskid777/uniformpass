# UniformPass — Seller Manage & Lost-Link Recovery UX Study
Design study, not a bug hunt. Ideal journeys designed from scratch within fixed constraints: no accounts, no login, secret manage-link (token in URL) is the only ownership proof, token also cached in localStorage on the posting device so "My Listings" + a "Manage your listing" button appear automatically there. Contact-form fallback exists but should be the last resort, not the primary recovery path.

---

## Seller — Needs to Edit / Mark Sold / Relist
**Goal:** Get to the manage screen for one specific listing in the fewest possible taps and either flip its status to sold, tweak the price, or relist it — before more buyers waste their time messaging about something that's gone.
**Mental state / constraints:** Task-locked and mildly annoyed — this is almost always triggered by an incoming text ("still available?") for an item they already sold. On the same phone they posted from, most of the time. Zero patience for re-finding the listing by browsing the public grid; wants a direct route from "I just sold this" to "done" in under 10 seconds. Will abandon and let stale listings rot if the path takes more than a couple of taps, which directly hurts marketplace trust (buyers messaging dead listings).
**Ideal entry point:** A persistent "My Listings" affordance in the site header/nav (badge-counted if any are live), reachable from any page, plus a direct "Manage your listing" button injected on the listing's own public detail page when localStorage has a matching token for that ID. The confirmation screen shown right after posting should also deep-link here, and the manage link itself (bookmarked or from a saved text/email) is a valid direct entry.
**Ideal click-path (numbered, ≤7 steps):**
1. Tap "My Listings" in the header (or the "Manage your listing" button on the item's public page, or a saved manage link).
2. See a compact list of this device's listings — status badge (Live / Sold / Pending) and thumbnail per row, sold items visually deprioritized but not hidden.
3. Tap the target listing row.
4. Land directly on its manage screen — status, price, and details all visible and editable on one screen, no sub-navigation.
5. Tap "Mark as Sold" (single button, no modal for the common case) — status flips instantly with inline confirmation.
6. (If relisting instead) Tap "Relist" on a Sold item — restores it to Live, optionally prompting a fresh "posted" timestamp so it doesn't look stale.
7. (If price/details) Tap "Edit," change the field inline, tap "Save" — return to manage view with the change reflected immediately.
**The ONE primary CTA per key screen:** My Listings list → tap a row (implicit "Manage this listing"). Manage screen (default/Live state) → "Mark as Sold." Manage screen (Sold state) → "Relist." Edit sub-view → "Save Changes."
**Trust / clarity reinforcement points:** Manage screen always shows the listing exactly as buyers see it (thumbnail + title) so there's no ambiguity about which item is being edited; every action (sold, relisted, price changed) gets an immediate, unmissable inline confirmation ("Marked as sold — buyers will now see this as unavailable") rather than a silent save; a small "last updated" timestamp on the manage screen and on My Listings so the seller can confirm a change actually took; sold items get a clear visual state (grayed card, "SOLD" ribbon) both in My Listings and on the public detail page so the seller trusts the status actually suppresses buyer contact.
**Components this journey demands (esp. manage surface, mark-sold, relist, account-less link recovery):** Header-level "My Listings" nav item reading localStorage token list, badge-counted by live listings; manage screen as a single-page edit surface (status + price + details, no multi-step wizard); one-tap "Mark as Sold" that also auto-hides/greys the listing from default browse results and disables its contact CTAs for buyers; "Relist" action that un-sells and refreshes recency signal (ties into the urgent-buyer persona's "listed X ago" sort); inline field editing with autosave-or-explicit-save pattern (not a full form re-navigate); toast/banner confirmation component for every mutating action.
**Friction to eliminate:** Having to hunt for the listing on the public browse grid to find the manage entry point; multi-screen confirmation modals for a routine "mark sold" action; edits that require re-entering fields that didn't change; any moment where the seller can't tell whether their action actually saved; sold listings that still show a "Contact Seller" button to buyers because the status change didn't propagate everywhere it needs to.
**Top 3 non-negotiables:**
1. "Mark as Sold" is reachable in ≤2 taps from any entry point (header nav or the item's own page) and takes effect immediately, everywhere the listing is shown to buyers.
2. My Listings and the manage screen work with zero typing — everything is tap/toggle for the common actions (sold, relist, price bump).
3. Every mutating action gets an explicit, visible confirmation — no silent saves the seller has to trust blindly.

## Seller — Lost Their Link / On a New Device
**Goal:** Regain the ability to self-manage a listing they posted, without creating an account, after losing the manage link or switching devices/browsers.
**Mental state / constraints:** Frustrated and slightly suspicious — they did nothing wrong (browser data cleared, new phone, using a laptop now) but the system is treating them like a stranger to their own listing. Doesn't want to relearn a new system or wait on a human; wants the fastest self-serve path back to the manage screen, and only wants to fall back to "email the operator" if self-serve genuinely can't work (e.g., they never gave any contact info at all). This is the exact seam where "no accounts" collides with "you should be able to prove you own this."
**Ideal entry point:** A visible "Lost your manage link?" affordance in two places — (1) the public listing detail page itself, right next to where "Manage your listing" would show if the token were present, and (2) a standalone `/recover` page linked from the footer and from My Listings' empty state ("No listings on this device — lost a link? Recover it"). The recovery should never require the seller to already know their listing ID.
**Ideal click-path (numbered, ≤7 steps):**
1. Tap "Lost your manage link?" from the listing page or the site footer/`/recover`.
2. Enter the contact info captured at post time (phone or email — whichever they used) into a single field.
3. Submit — system looks up all listings tied to that contact info (no confirmation of what it found yet, to avoid leaking data to a guesser).
4. See a neutral confirmation screen: "If we found listings tied to that contact, we've sent manage links to it." (Same message whether zero, one, or many matches — prevents enumeration.)
5. Seller receives an email or SMS with manage link(s) — one link per active listing tied to that contact, clearly labeled by item/photo.
6. Tap the link on the new device — lands directly on that listing's manage screen, and the token is now cached in this device's localStorage too, so My Listings works here going forward.
7. (No match / contact info wasn't captured at post time) Screen offers the Contact form fallback as an explicit secondary option, pre-filled with whatever identifying details the seller can provide (school, item, approximate post date) for the operator to manually verify and resolve.
**The ONE primary CTA per key screen:** Listing detail (no token present) → "Lost your manage link?" `/recover` entry → "Send me my manage links." Confirmation screen → (no CTA needed — informational; secondary link to "Try again" or "Contact us instead"). Delivered email/SMS → "Manage this listing" (per item).
**Trust / clarity reinforcement points:** The neutral "if we found a match" messaging on submission (never confirms or denies whether an email/phone is in the system, so it can't be used to snoop on other sellers' listings); the recovery link/SMS explicitly names the item and shows its photo so the seller isn't blindly trusting a link; links are time-limited or single-purpose (scoped only to re-establishing manage access, not a general login) so the seller understands this isn't secretly becoming an account; a short line of copy on `/recover` explaining *why* this exists ("We don't use accounts or passwords — so we send your manage link to the contact you gave us when you posted") turns the friction into a legible tradeoff instead of a mystery.
**Components this journey demands (esp. manage surface, mark-sold, relist, account-less link recovery):** Contact-info capture at post time (already required for buyer contact — reused, not a new field) that doubles as the recovery lookup key; a `/recover` endpoint that looks up all listings by phone/email match and dispatches manage links via email (transactional send) and/or SMS; rate-limiting / neutral-response pattern on the lookup to prevent contact-info enumeration; multi-listing digest format (one email/SMS listing all active items tied to that contact, not one send per item) for sellers with multiple postings; on-click-through, silent re-hydration of localStorage with the recovered token so the new device becomes a fully working "home device" going forward; Contact-form fallback retained and clearly positioned as the path for sellers whose contact info wasn't captured, changed, or who can't access it anymore.
**Friction to eliminate:** Requiring the seller to remember or locate their listing ID/URL to start recovery; any recovery flow that requires them to already have the lost link (circular); confirming/denying matches in a way that leaks whether a given email/phone has listings (privacy/enumeration risk); sending a bare, unlabeled link that looks like phishing; forcing every recovery case through the manual Contact-form/operator queue when self-serve is possible; losing the recovered device's access again next time (must re-cache token so this doesn't repeat).
**Top 3 non-negotiables:**
1. Recovery is entirely self-serve via contact info already captured at post time — the operator/Contact-form path is the exception case, not the default, and should visibly shrink in volume as this ships.
2. Lookup responses are neutral regardless of match outcome, to avoid turning "lost my link" into a way to probe other people's contact info or listings.
3. Successful recovery re-hydrates localStorage on the new device, so the seller doesn't fall into the same hole again on their next edit.

## The Tension & Recommended Stance
No-accounts is the product's core trust promise (frictionless post, no password, no data retention beyond what's needed) — but it means "prove you own this listing" has exactly one artifact (the token) and exactly one durable home (one device's localStorage) unless something else is captured. That's a single point of failure by design.

**Recommendation:** Make contact-info capture at post time mandatory, not optional (it's already collected for buyer-to-seller contact — extend its purpose to double as the recovery key, don't add a new field). Ship `/recover` as email/SMS-delivered magic links scoped strictly to manage access, not a general session/login — this keeps the product honestly account-less (no persistent identity, no password, no cross-listing profile) while closing the "new device" gap for the ~90%+ case where the seller used real contact info. Keep the Contact-form/operator path alive permanently as the floor for the remaining edge cases (fake/typo'd contact info, contact info seller no longer controls) — it should never be removed, just demoted from primary to last-resort. This is the honest tradeoff: no accounts means no persistent identity, but it doesn't have to mean no recovery — recovery just has to be re-derived per request instead of remembered.
